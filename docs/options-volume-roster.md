# Options-Volume Roster — Architecture Reference

This document is the single authoritative reference for the options-volume
roster: where it comes from, what it powers, how to cut it for various chart
filters, and how the planned three-tier architecture (anchor / dynamic
tail / dampening) is meant to behave. Agents working on /heatmap, /scan,
/earnings, or any new surface that wants to scope itself to "names a vol
trader cares about" should read this end to end before changing the JSON
or proposing a new threshold. This doc is self-contained and cold-readable
— do not assume prior conversation context.

## What the roster is

The OV roster is a list of ~250 single-name US-listed equities (stocks
and ADRs) ranked by trailing US options contract volume. It is **not**
the SP500. Many roster names are not SP500 members (NVDA-adjacent semis,
crypto-mining cluster, recent IPOs, ADRs); many SP500 names are not in
the roster (low-OV utilities, REITs, consumer staples). The SP500 roster
is a separate file (`src/data/sp500-roster.json`, generated by
`scripts/backfill/sp500-roster.mjs`) and is used for sector-classification
joins, not for universe scoping.

The OV roster exists because vol traders care about the names they
actually trade — names with deep, liquid options markets — and the SP500
universe diverges materially from that set.

## Files

- **`src/data/options-volume-roster.json`** — the roster itself,
  consumed by /heatmap, /earnings, /scan, and any new vol-focused surface.
- **`scripts/backfill/options-volume-roster.mjs`** — the generator. Reads
  a Barchart CSV from `C:\sheets\`, joins to GICS sectors, writes the
  JSON. Header docstring (lines 1-48) is the operational reference for
  running it.
- **`docs/earnings-data-roadmap.md`** — strategic data roadmap for the
  /earnings lab; references this doc for roster shape decisions.

## Current data source (manual)

A Barchart "stocks screener" CSV that Eric exports manually and drops
into `C:\sheets\stocks-screener-MM-DD-YYYY.csv`. CSV columns: `Symbol`,
`Name`, `Exchange`, `Options Vol`. The Options Vol column is **spot-day
contract count**, not a rolling average. This matters — see the
"Earnings-spike volatility problem" section below.

The script auto-discovers the most recent matching file in
`C:\sheets\`, so you do not need to pass a path argument.

## Planned automated source (TBD)

Two paths to replace the manual Barchart pull:

1. **Massive grouped options aggregates** (`/v2/aggs/grouped/locale/us/market/options/{date}`).
   Returns one row per options contract per date with daily volume.
   Aggregating by underlying ticker gives per-ticker daily volume in a
   single API call. Roll over 30 days, sort descending, take top N. This
   is the right tool for *universe discovery* (replacing Barchart). Tier
   gating is uncertain — Massive Stocks Starter (current subscription)
   may not include options aggregates. To verify, run:
   ```
   curl -H "Authorization: Bearer $MASSIVE_API_KEY" \
     "https://api.polygon.io/v2/aggs/grouped/locale/us/market/options/<recent-date>"
   ```
   200 = entitled. 401/403 with NOT_AUTHORIZED = tier upgrade required.

2. **ThetaData Options Standard** (`option/history/eod` with `expiration=*`
   per ticker per date). Returns every listed contract for a ticker on a
   date, including volume. Already entitled. Good for *refreshing volume
   on a known ticker list*, NOT for discovering an unknown universe
   (chicken-and-egg — you need a ticker list before you can query it).

Until path 1 is verified entitled, the manual Barchart pull is
authoritative.

## Distribution shape (2026-04-26 pull, n=250)

Heavy power-law decay. NVDA solo at 6.75M daily contracts; the bottom
of the roster (KHC) is 17.5k. Natural breakpoints at clean
round-number thresholds:

| Threshold (daily contracts) | Names through this threshold | Last name in band |
|---|---|---|
| ≥ 1,000,000 | 6 | AAPL |
| ≥ 500,000 | 12 | GOOGL |
| ≥ 250,000 | 25 | MRVL |
| ≥ 100,000 | 62 | LRCX |
| ≥ 50,000 | 112 | NEM |
| ≥ 25,000 | 200 | CAT |
| ≥ 17,000 | 250 (full roster) | KHC |

These are the canonical bucket boundaries. New consumers should adopt
**volume thresholds, not rank-based "Top N"**. Rank-based filters are
opaque to users and unstable across refreshes; threshold filters are
self-explanatory and stable.

## Where to cut the roster — by purpose

| Use case | Cutoff | Approximate name count | Why |
|---|---|---|---|
| /earnings scatter chart, default | OV ≥ 100k | ~62 | Liquid enough that ATM straddles are well-defined; chart density stays under ~80 dots even on peak earnings weeks. |
| /earnings scatter, mega-cap-only | OV ≥ 250k | ~25 | For peak weeks where 60+ dots is too dense. |
| /earnings calendar grid | OV ≥ 25k | ~200 | Wide-net comprehensive view; the grid layout absorbs density. |
| /heatmap | full roster (250) | 250 | Visual density is the constraint, not liquidity. |
| Anchor list (hardcoded core) | rank 1-50 candidate pool, then editorial filter | ~30-35 | See "Anchor list" section below. |

When in doubt, prefer threshold-based cutoffs over rank-based. The user
should be able to read the filter pill ("OV ≥ 100k") and know what they
are looking at without having to reason about "what does Top 100 mean
this week vs. last week."

## Three-tier roster architecture

The roster is conceptually three layers. Layer 1 is editorial; Layers
2-3 are mathematical. **As of 2026-04-26, only the flat unsorted-tier
list is implemented.** The schema migration to the three-tier model is
documented below but not yet wired in.

### Layer 1 — Anchor (editorial, hardcoded)

~30-40 names that are structurally always at the top regardless of the
current week's data. Pinned via a `tier: "anchor"` and `pinned: true`
flag in the JSON; the generator script *never* drops these regardless of
their current rank in the data feed. This is editorial — Eric curates
the anchor list the way he curates the `MANUAL_SECTORS` map at the
bottom of `options-volume-roster.mjs`.

**Promotion criteria:** a watchlist name (see below) survives 2-3
quarterly refreshes near the top of the rolling-window ranking before
being promoted to anchor.

**Demotion criteria:** an anchor name's options market materially dies
(e.g., post-acquisition delisting, sustained > 6 months below the rolling
top 100). Editorial decision, not automatic.

### Layer 2 — Dynamic tail (data-derived, smoothed)

Names below the anchor cutoff, derived from a **30-day rolling-average
options-volume ranking**. Use Massive grouped aggregates (if entitled)
or ThetaData EOD over a known superset. Total roster size stays at ~250.

The tail is the layer that churns. Single-week ranking shifts in the
tail are normal and expected.

### Layer 3 — Mid-band dampening

Two filters applied when computing the tail's 30-day rolling-average
ranking:

1. **Earnings quarantine.** When computing the rolling average for a
   ticker, exclude the trailing 5 trading days around each of its
   earnings reports. The earnings calendar is already in the /earnings
   lab (EarningsWhispers, scraped server-side by
   `netlify/functions/earnings.mjs`); the roster generator joins to it
   and skips the spike window. This kills the "AAPL reports → MARA gets
   pushed off the roster" problem at the source.

2. **Hysteresis.** A name has to clear the threshold for **2 consecutive
   weekly refreshes** to enter the tail, and stay below for 2
   consecutive to leave. Stops one-week shocks from churning membership.
   Implemented via a `consecutiveAppearances` counter persisted in the
   JSON across refreshes.

Both filters are mathematical and require no editorial maintenance.

## Earnings-spike volatility problem

Barchart's "Options Vol" column is **spot-day contract count**. A name
reporting earnings does 5-30x its normal contract count for 24-72 hours
and then falls back to baseline. If the CSV is pulled the day after
earnings, that name appears at the top of the screener for one week of
roster freshness, then drops back at next refresh.

This is the volatility the three-tier architecture exists to solve:

- **Anchor names** are immune by construction (pinned).
- **Tail names** are smoothed by the 30-day rolling window + earnings
  quarantine + hysteresis.
- **Today (without the architecture):** the roster carries spike-driven
  noise in positions ~30-200, visible in current top-50 names like CRWV,
  POET, ONDS, BMNR, SNDK whose presence at this rank is unrepresentative
  of their durable position.

When the three-tier architecture ships, the spike noise drops out
automatically.

## When to revisit the architecture

- **Massive options-aggregates entitlement confirmed.** Wire up
  `/v2/aggs/grouped/locale/us/market/options/{date}` as the canonical
  tail-refresh source; deprecate the Barchart manual pull. Keep the
  manual pull as a fallback for one quarter, then remove.
- **Market-cap weighting wanted on /heatmap or any consumer.** Add a
  `marketCap` field per ticker via Massive
  `/v3/reference/tickers/{ticker}` (Stocks Starter already covers it).
  See `docs/earnings-data-roadmap.md` §1 for the full plan. Do NOT
  blend market cap into OV ranking — they are orthogonal axes.
- **Anchor watchlist outgrows manual maintenance** (~10+ names in
  the watchlist tier at any time). At that point, formalize the
  promotion criteria into a script that suggests promotions based on
  refresh-history data, with editorial sign-off retained.

## What NOT to do

- **Do not promote a watchlist name to anchor based on a single
  quarter's data.** The whole point of the watchlist is to survive
  multiple quarters before being promoted.
- **Do not blend market cap into the OV ranking.** They are orthogonal
  axes. Market cap belongs in a separate `marketCap` field per ticker.
- **Do not replace OV ranking with market cap.** Eric explicitly
  rejected MAG7-dominance for /heatmap (see the auto-memory entry
  `project_heatmap_design`). The OV roster exists precisely because
  vol traders care about names that may not be top-mcap (PLTR, MARA,
  HIMS, IREN, MSTR, etc.).
- **Do not introduce rank-based filter UIs for new surfaces.** Use
  threshold-based cutoffs (`OV ≥ 100k`) so the user can reason about
  the filter and the behavior stays stable across refreshes.
- **Do not tie the tail-refresh cadence to the anchor-review cadence.**
  Anchor review is quarterly editorial; tail refresh is weekly
  automated. Coupling them defeats the point of the layering.
- **Do not silently classify unknown tickers into "Other" at the
  sector-join step.** The script's existing behavior is to log to
  stderr and exit 1 if a ticker is missing both from the SP500 CSV and
  from the `MANUAL_SECTORS` override map; preserve that. Better to
  catch missing classifications at roster-generation time than to
  render the heatmap with a giant "Other" region.
